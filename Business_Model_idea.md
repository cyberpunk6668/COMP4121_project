电子设备上门维修平台开发详细文档
1. 项目概述
本项目旨在构建一个美观、易用的电子设备上门维修服务平台。用户可以通过网站提交设备维修请求，选择维修服务类型、预约时间和地址，支付费用后，系统自动匹配附近的维修工程师上门服务。平台还提供工程师管理、订单跟踪、评价反馈等功能。

目标用户：普通用户（需要维修设备）、维修工程师（接单并提供服务）、平台管理员（管理订单、用户、工程师等）。

2. 功能需求
2.1 用户端（前台）
首页：展示平台介绍、热门维修服务、优惠活动、工程师风采、用户评价等。

设备选择：按设备类型（手机、平板、电脑、相机等）筛选，展示常见故障和维修方案。

服务详情：点击具体维修项，显示维修价格、耗时、服务说明、包含配件等。

预约下单：填写设备信息（型号、问题描述）、选择上门地址、预约时间段、选择支付方式（微信/支付宝/银行卡）。

订单管理：查看订单状态（待支付、待分配、工程师已接单、上门中、已完成、已取消），可取消订单、评价已完成订单。

个人中心：管理个人信息、地址簿、优惠券、收藏、消息通知。

用户评价：对完成的订单进行评分和文字评价。

在线客服：与客服进行实时聊天（可集成第三方客服工具）。

2.2 工程师端（需登录）
接单池：查看附近待分配的订单，可抢单或由系统指派。

我的订单：查看已接订单，包括用户信息、地址、服务内容，可联系用户、开始服务、完成服务。

工作台：查看今日任务、收入统计、提现申请。

个人资料：维护个人信息、技能证书、服务区域等。

2.3 管理后台（管理员）
仪表盘：展示平台总订单数、收入、用户数、工程师数等统计数据。

用户管理：查看、禁用/启用用户，重置密码。

工程师管理：审核工程师入驻申请，设置服务区域、上下线状态，查看工程师业绩。

订单管理：查看所有订单，可手动分配工程师，处理纠纷订单，查看订单详情。

服务管理：维护设备类型、维修项目、价格、配件信息。

营销管理：发布优惠券、活动banner、公告。

财务管理：查看交易流水，处理工程师提现申请。

系统设置：基础参数配置（如提现门槛、客服电话等）。

3. 技术栈建议
为了快速开发并拥有好看的UI，建议采用以下技术栈：

3.1 前端
框架：React 18 + TypeScript (或 Vue 3 + Composition API)

UI组件库：Ant Design (React版) 或 Element Plus (Vue版)，提供大量现成美观组件。

状态管理：Redux Toolkit (React) / Pinia (Vue)

路由：React Router 6 / Vue Router 4

HTTP请求：axios

地图服务：集成高德地图/腾讯地图API，用于地址选择和工程师位置展示。

打包工具：Vite 或 Webpack

CSS预处理器：Sass/Less

移动端适配：可考虑响应式设计或单独开发移动端（本项目以PC为主，但需适配平板）

3.2 后端
语言框架：Node.js + Express (或 Python Django/Flask，Java Spring Boot)。推荐Node.js，便于前端同学上手。

数据库：MySQL 8.0 (关系型数据) + Redis (缓存、会话、消息队列)

ORM：Sequelize (Node.js) / Prisma

认证：JWT (JSON Web Token) 或 Session + Redis

文件存储：OSS (阿里云OSS/七牛云) 用于上传用户头像、设备图片等

支付：微信支付、支付宝支付接口集成

短信服务：阿里云/腾讯云短信，用于发送验证码、通知

地图服务：使用高德/腾讯地图API实现距离计算、地址解析

即时通讯：可集成融云/环信或WebSocket自建简单客服

3.3 部署
服务器：Linux (CentOS/Ubuntu)

Web服务器：Nginx (反向代理)

容器化：Docker + Docker Compose 便于部署

CI/CD：GitHub Actions / GitLab CI

5. API接口设计
采用RESTful风格，返回JSON格式。所有接口除登录注册外均需携带JWT（放在Authorization头）。以下列出主要接口：

5.1 用户认证
POST /api/auth/register 注册（手机号+验证码）

POST /api/auth/login 登录（手机号+密码/验证码）

POST /api/auth/logout 登出

POST /api/auth/refresh-token 刷新token

GET /api/auth/profile 获取个人信息

PUT /api/auth/profile 更新个人信息

5.2 地址管理
GET /api/addresses 获取地址列表

POST /api/addresses 新增地址

PUT /api/addresses/:id 修改地址

DELETE /api/addresses/:id 删除地址

PATCH /api/addresses/:id/default 设为默认

5.3 设备与服务
GET /api/device-types 获取设备类型列表

GET /api/device-types/:id/items 获取某设备下的维修项目

GET /api/repair-items/:id 获取维修项目详情

5.4 订单流程
POST /api/orders 创建订单（预下单，可传优惠券ID）

GET /api/orders 获取当前用户的订单列表（支持状态过滤）

GET /api/orders/:id 获取订单详情

PUT /api/orders/:id/cancel 取消订单

POST /api/orders/:id/pay 支付订单（调用支付接口，返回支付参数）

GET /api/orders/:id/pay/result 查询支付结果

POST /api/orders/:id/review 评价订单

5.5 工程师接单相关
GET /api/engineer/orders/pending 待接单列表（按距离排序，需传经纬度）

POST /api/engineer/orders/:id/accept 接受订单

PUT /api/engineer/orders/:id/start 开始服务

PUT /api/engineer/orders/:id/complete 完成服务

GET /api/engineer/orders 工程师自己的订单列表

GET /api/engineer/stats 工程师统计（今日订单、收入等）

5.6 管理员接口（需管理员权限）
GET /api/admin/orders 所有订单列表

PUT /api/admin/orders/:id/assign 分配工程师

GET /api/admin/engineers 工程师列表

PUT /api/admin/engineers/:id/status 修改工程师状态

等等

5.7 其他
GET /api/coupons 获取当前用户可用的优惠券

POST /api/upload 上传图片（返回URL）

6. 前端UI设计
需要“好看的UI界面”，因此要注重视觉设计。建议使用现成的UI组件库并定制主题色。以下是主要页面设计概要：

6.1 首页
顶部导航：Logo、搜索框、登录/注册、个人中心入口。

Banner轮播：展示活动广告。

快捷入口：设备类型图标（手机、电脑、平板等），点击跳转至对应服务列表。

热门服务：展示几个热门维修项目，带价格和预约按钮。

服务流程：图文展示下单-上门-维修-评价流程。

工程师风采：展示部分优秀工程师（头像、姓名、评分、接单数）。

用户评价：轮播显示好评内容。

底部：关于我们、联系方式、友情链接等。

6.2 服务列表页
左侧筛选：按设备类型、维修项目、价格区间筛选。

右侧列表：卡片形式展示维修项目，包含设备图、名称、价格、预约按钮。支持排序（价格、销量）。

6.3 服务详情页
大图展示：维修示意图。

服务标题：名称、价格、销量、评分。

服务说明：包含配件、服务流程、保修政策等。

用户评价：展示该项目的用户评价。

预约按钮：点击进入下单页。

6.4 下单页（多步骤表单）
第一步：选择设备 选择设备类型、填写型号、描述问题。

第二步：选择服务 选择维修项目（如果有多个可选，比如屏幕维修可能有不同品质的屏）。

第三步：预约信息 选择地址（可新增）、选择上门时间（日期+时间段）。

第四步：确认订单 展示订单详情、使用优惠券、选择支付方式。

第五步：提交支付 显示支付二维码或跳转支付。

6.5 个人中心
侧边栏：我的订单、地址管理、优惠券、个人信息、消息、设置。

主要内容区：展示当前用户信息、待办订单卡片。

我的订单：按状态标签分类（待支付、待分配、服务中、待评价、已完成），点击进入详情。

地址管理：列表显示地址，可编辑删除。

优惠券：展示未使用/已使用/已过期优惠券。

6.6 工程师端（类似移动端或PC工作台）
仪表盘：今日任务数、收入、评价。

8.2 核心功能实现细节
8.2.1 用户认证
注册：发送短信验证码，验证后写入数据库，密码加密存储。

登录：验证手机号密码，生成JWT，返回token和用户信息。

使用express-jwt中间件保护需要认证的路由。

8.2.2 订单创建
接收参数：repair_item_id, device_model, problem_desc, address_id, appointment_time, coupon_id等。

验证维修项目是否存在，获取价格。

验证地址属于当前用户。

计算优惠（若有优惠券）。

生成唯一订单号（规则如：年月日+随机数）。

写入orders表，状态为待支付。

返回订单信息。

8.2.3 支付集成
以微信支付为例，需申请商户号，配置API密钥。

创建支付时，调用统一下单接口，获取prepay_id，返回给前端签名参数。

前端调起支付，成功后后端接收异步通知更新订单状态为已支付，并进入待分配。

需处理支付回调幂等性。

8.2.4 工程师分配
用户支付成功后，订单状态变为“待分配”。

可以有两种模式：抢单模式（工程师主动抢）或系统指派（根据距离、评分、空闲状态）。

抢单：工程师通过接口获取附近待分配订单，点击抢单，更新订单engineer_id和状态为“待上门”。

系统指派：定时任务扫描待分配订单，计算附近工程师，推送通知，接受后锁定。

8.2.5 工程师服务流程
工程师上门后点击“开始服务”，状态变为“服务中”。

维修完成后点击“完成服务”，状态变为“待评价”，并记录完成时间。

用户可评价，完成后状态变为“已完成”。

8.2.6 地图服务
地址选择：前端调用高德地图API，获取经纬度，保存到地址表。

附近工程师搜索：根据用户地址经纬度，计算距离，筛选一定范围内的工程师（可使用MySQL空间函数或GeoHash）。

8.2.7 消息通知
订单状态变化时，通过WebSocket或短信通知相关方。

可集成极光推送或自己用Socket.IO实现。

11. 其他注意事项
11.1 安全性
防止SQL注入：使用ORM或参数化查询。

防止XSS：对用户输入进行转义。

防止CSRF：使用JWT无状态token，设置HttpOnly Cookie可减少风险，但前后端分离通常将token存localStorage，需注意XSS攻击。可考虑将token存于HttpOnly Cookie中，但需处理CSRF。

接口限流：使用express-rate-limit防止暴力破解。

敏感信息加密：如身份证号。

11.2 性能优化
前端代码分割（React.lazy），减少首屏加载时间。

图片懒加载。

后端数据库索引（订单号、用户ID、状态等）。

使用Redis缓存常用数据（如设备类型列表）。
